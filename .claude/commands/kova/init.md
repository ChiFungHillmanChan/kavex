# /kova:init
# Scaffold a new PRD file for the Kova Smart Loop.
# Creates a properly formatted PRD that /kova:loop can parse.
#
# Usage:
#   /kova:init                     → interactive, asks what you want to build
#   /kova:init auth-system         → scaffold PRD for "auth-system"
#   /kova:init docs/prd-auth.md    → scaffold at specific path

You are scaffolding a PRD (Product Requirements Document) for the Kova Smart Loop. Follow these steps.

---

## Step 1: Determine what to scaffold

The user argument is: `$ARGUMENTS`

**If `$ARGUMENTS` is empty:**
Ask the user:
```
What feature are you building? Give me a short name or description.
Examples:
  - auth-system
  - payment-integration
  - dashboard-redesign
  - api-v2-endpoints
```
Wait for their response.

**If `$ARGUMENTS` looks like a file path** (contains `/` or ends in `.md` or `.json`):
- Use it as the output path
- Extract the feature name from the filename (e.g., `docs/prd-auth.md` → "auth")
- Ask: "What should this PRD cover? Describe the feature briefly."
- Wait for their response.

**If `$ARGUMENTS` is a feature name** (no path separators):
- Use `docs/prd-<argument>.md` as the output path (create `docs/` if needed)
- The feature name is the argument itself

## Step 2: Explore the codebase for context

Before writing the PRD, quickly understand the project:

1. Read `README.md` if it exists (for project overview)
2. Read `.claude/CLAUDE.md` for tech stack and conventions
3. Check the project structure:
   ```bash
   ls -la
   ```
4. If there's a `package.json`, `pyproject.toml`, `go.mod`, etc., check what's already built

This gives you context to write PRD items that fit the existing codebase.

## Step 3: Generate the PRD

Ask the user:
```
How detailed should the PRD be?

  1. Quick    — 3-5 high-level items (I'll figure out the details)
  2. Detailed — 8-15 specific items with descriptions
  3. Custom   — I'll describe exactly what I want
```

Wait for their choice.

### For "Quick" or "Detailed":

Based on the feature name, codebase context, and detail level, generate a PRD. Think about:
- What already exists in the codebase?
- What needs to be built from scratch?
- What's the logical implementation order? (schema first, then API, then UI, then tests)
- What are the edge cases?

### For "Custom":

Ask: "Describe the items you want implemented. I'll format them into a PRD."
Wait for their description, then structure it.

## Step 4: Write the PRD file

Create the file at the determined path. Use this format:

```markdown
# PRD: <Feature Name>

> Generated by Kova Init — <date>
> Run with: `/kova:loop <this-file-path>`

## Overview

<1-2 sentence description of the feature>

## Items

- [ ] <Item 1 — typically foundation/schema/setup>
- [ ] <Item 2 — core functionality>
- [ ] <Item 3 — secondary functionality>
- [ ] <Item 4 — edge cases and error handling>
- [ ] <Item 5 — tests and validation>
...

## Notes

- Tech stack: <detected from codebase>
- Dependencies: <any external packages needed>
- Assumptions: <any assumptions made>
```

Rules for good PRD items:
- **Each item should be independently implementable** — Item 2 can build on Item 1's output, but shouldn't require Item 3
- **Items should be ordered** — foundations first (schema, types, config), then core logic, then edge cases, then polish
- **Items should be specific** — "Implement POST /api/signup with email validation, password hashing, and JWT response" NOT "Add signup"
- **Items should be testable** — Claude needs to know when an item is "done"
- **Include test items** — at least one item should be dedicated to integration/e2e tests
- **5-15 items is ideal** — fewer than 5 means items are too big; more than 15 means too granular

## Step 5: Show the result and confirm

Display:
```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 PRD CREATED
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 File: <path>
 Items: <N> pending
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

<show the items list>

Edit the file to adjust items, then run:
  /kova:loop <path>
```

Then ask: **"Want to start the loop now, or edit the PRD first?"**

- If they say "go" / "start" / "yes" → immediately invoke `/kova:loop <path>`
- If they say "edit" / "no" / "later" → stop and let them edit

---

## JSON format alternative

If the user specifically asks for JSON format, use:

```json
{
  "name": "<Feature Name>",
  "generated": "<date>",
  "items": [
    {
      "title": "<Item 1>",
      "description": "<Detail about item 1>"
    },
    {
      "title": "<Item 2>",
      "description": "<Detail about item 2>"
    }
  ]
}
```

But default to markdown — it's easier to read and edit.
